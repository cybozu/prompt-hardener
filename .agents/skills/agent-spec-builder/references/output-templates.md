# Output Templates

Templates for the 3 output files generated by agent-spec-builder.

---

## 1. agent_spec.yaml

### Field Ordering (follow this order for all types)

```yaml
version: "1.0"
type: <type>
name: "<name>"
description: "<description>"

system_prompt: |
  <system prompt text>

# messages: (optional)
#   - role: user
#     content: "..."
#   - role: assistant
#     content: "..."

provider:
  api: <api>
  model: <model>
  # region: <region>    # bedrock only
  # profile: <profile>  # bedrock only

# --- Type-conditional sections below ---

# tools: (agent, mcp-agent)
# data_sources: (rag, agent, mcp-agent)
# mcp_servers: (mcp-agent)
# policies: (all types, recommended for agent/mcp-agent)

user_input_description: "<description>"
has_persistent_memory: "<true|false|unknown>"
scope: "<single_user|shared_workspace|multi_tenant|unknown>"
```

### Tool Entry Template

```yaml
tools:
  - name: <tool_name>
    description: "<what this tool does>"
    parameters:
      type: object
      properties:
        <param_name>:
          type: <param_type>
          description: "<param description>"
      required: [<required_params>]
    effect: <read|write|delete|external_send|unknown>
    impact: <low|medium|high|unknown>
    execution_identity: <user|service|mixed|unknown>
```

### YAML Comment Annotations

When a value is inferred or unknown, add inline comments:

```yaml
# Confidence annotations:
effect: write          # confidence: high — explicit HTTP PUT in implementation
effect: read           # confidence: medium — inferred from "get_" prefix
effect: unknown        # TODO: Set to read/write/delete/external_send → enables TOOL-001, TOOL-006

# Unknown value annotations (always include the rule it would enable):
impact: unknown        # TODO: Set to low/medium/high → enables TOOL-003 (critical)
execution_identity: unknown  # TODO: Set to user/service/mixed → enables TOOL-004 (high)

# Safe default annotations:
trust_level: untrusted  # Safe default — set to "trusted" if you control this source
```

### Evidence Source Formats

- **Code reference**: `src/agent.py:45 — @tool decorator found`
- **Import detection**: `from openai import OpenAI at src/main.py:3`
- **Pattern match**: `Inferred from "get_" prefix in tool name`
- **User response**: `User stated: "openai / gpt-4o"`
- **Safe default**: `Default value — could not determine from available information`
- **File content**: `Content of prompts/system.txt`

## 3. open_questions.md

```markdown
# Open Questions

Items below could not be determined automatically. Resolving them enables additional
security analysis rules in Prompt Hardener.

## Critical Priority

These fields enable **critical-severity** rules. Resolving them is strongly recommended.

### tools[<N>].impact
- **Current value**: `unknown`
- **Why it matters**: Setting to `low`/`medium`/`high` enables **TOOL-003** (critical) —
  checks that high-impact tools have escalation coverage.
- **How to resolve**: Consider the worst-case scenario if this tool is misused.
  If it can cause data loss, financial impact, or security breach → `high`.
  If it can modify configuration or non-critical data → `medium`.
  If it only reads data → `low`.

### tools[<N>].effect
- **Current value**: `unknown`
- **Why it matters**: Setting to `read`/`write`/`delete`/`external_send` enables
  **TOOL-006** (critical) — detects exfiltration risk when combined with confidential data,
  and **TOOL-001** (high) — checks deny/escalation coverage for sensitive tools.
- **How to resolve**: What side effect does this tool have?
  `read` = only retrieves data, `write` = creates/modifies data,
  `delete` = removes data, `external_send` = sends data outside the system.

## High Priority

These fields enable **high-severity** rules.

### tools[<N>].execution_identity
- **Current value**: `unknown`
- **Why it matters**: Setting to `user`/`service`/`mixed` enables **TOOL-004** (high) —
  checks that service-identity tools have escalation coverage.
- **How to resolve**: Does this tool run with the end user's permissions (`user`),
  a service account (`service`), or both depending on context (`mixed`)?

### policies.escalation_rules
- **Current value**: not defined
- **Why it matters**: Enables **TOOL-001**, **TOOL-003**, **TOOL-004**, **ARCH-001** (high) —
  verifies sensitive/high-impact/service-identity tools have human oversight.
- **How to resolve**: Define conditions under which the agent should pause and
  ask for human confirmation. Example:
  ```yaml
  escalation_rules:
    - condition: "User requests account deletion"
      action: "Ask for explicit confirmation"
  ```

### policies.denied_actions
- **Current value**: not defined
- **Why it matters**: Enables **TOOL-001**, **TOOL-003**, **TOOL-004** (high) —
  verifies dangerous actions are explicitly blocked.
- **How to resolve**: List actions the agent must never perform.

### has_persistent_memory
- **Current value**: `unknown`
- **Why it matters**: If `"true"`, enables **ARCH-005** (high) — checks for memory
  poisoning protection in the system prompt.
- **How to resolve**: Does your agent store conversation history or context
  that persists across user sessions?

### scope
- **Current value**: `unknown`
- **Why it matters**: If `"multi_tenant"`, enables **ARCH-006** (high) — checks for
  tenant isolation when sensitive tools are available.
- **How to resolve**: Does each agent instance serve one user, a team, or
  multiple organizations?

### data_sources[<N>].sensitivity
- **Current value**: `unknown`
- **Why it matters**: Setting to `confidential` enables **TOOL-005** (high) — checks
  data boundary coverage, and **TOOL-006** (critical) — detects exfiltration risk.
- **How to resolve**: Does this data source contain PII, financial data, or
  trade secrets (`confidential`)? Internal docs (`internal`)? Public content (`public`)?

## Medium Priority

These fields improve analysis quality but don't enable high/critical rules.

### user_input_description
- **Current value**: not defined
- **Why it matters**: Improves remediation quality by providing context about
  how users interact with the agent.
- **How to resolve**: Describe the input channel, e.g., "Free-form text via chat widget".

### policies.allowed_actions
- **Current value**: not defined
- **Why it matters**: Enables **TOOL-002** (medium) — checks that a tool allowlist exists.
- **How to resolve**: List the tools the agent is explicitly permitted to use.
```

### Notes on open_questions.md

- Only include fields that are actually `unknown` or missing in the generated spec
- Use actual tool indices (e.g., `tools[0]`, `tools[1]`) not placeholders
- Reference the actual tool names in the text for clarity
- Group by priority (Critical > High > Medium)
- Each entry must include: current value, why it matters (rule ID + severity), how to resolve
